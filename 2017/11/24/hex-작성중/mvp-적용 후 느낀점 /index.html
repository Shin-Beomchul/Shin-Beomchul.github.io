MVP Vs MVC

null 작성중 2017/11/24

本文于348天之前发表，文中内容可能已经过时。如有疑问，请在评论区留言。

1. 프로젝트 적용 후 느낀 점

[장점]

  • 구축된 이후로 유지 보수가 훨씬 쉽다.
  • 프로그램의 동작 흐름을 Presenter에서 확인 할 수 있다.
  • View 의 인터페이스만 보고도 어떤 기능을 하는지 짐작 할 수 있다.
  • View와 Model은 MainThread ,SubThread가 명확히 분리 되어 있다.
    : 자신이 작성하지 않은 코드나 여러명이서 작업한 코드가 거대해 지다보면 Ui Thread 에서 동작해야하는지, MainThread 에서 동작 해야하는지 모호해지는 경우가 있습니다. 대부분의 경우 경험상 추론이나 코드를 유심히 봐야 “아 이코드는 SubThread 에서 동작 해야하는구나” 라고 생각하며 코드를 수정 하는데 이런 불필요한 분석시간을 줄알 수 있다.
  • Model이 분리 되 있으므로 재사용 하기 쉽다.

  • 학습되지 않았을때의 경우 오랜 시간이 걸릴뿐, 익숙해진 후 View 인터페이스를 정의 할때 부터
    좀더 기능에 대한 1차적인 설계가 되서 오히려 코드가 복잡하고 거대해 질 경우 삽질 방지 효과가 뛰어남

    [단점]

  • MVP표준이 없다. 팀원이 MVP를 이해하고 초기 브리핑 없이 개발하게 되면 각자의 아키텍쳐로 개발 될 수있다.
  • 최초 프로젝트 구축 비용이 많이 든다.
  • 강제된 인터페이스가 아니므로 급한 일정앞에 모호성과 귀찮음과 타협하여 MVP 패턴과 타협하게 됨.

[모호성]
프로그래밍 패턴에서의 모호성은 예전 부터 존재해 왔습니다.
Spring처럼 강제화된 MVC 패턴 인터페이스를 가지지 않는 이상, 프로그래밍 패턴을 적용한다는 것은 코드를 개발하는 사람에 따라 MVC,MVP MVVM 등 패턴을 적용한다 하더라도 각자 혹은 팀의 속성이 반영된 패턴으로 개발 될수 밖에 없습니다. 예를 들어 MVP Model-Presenter-View를 요약하면 UI 나 화면은 View에서 처리하고, Model은 DB 나 API를 통해 데이터를 가져오는 역할을 가지며 Presenter는 이 두개의 레이어를 관리 하는 패턴 이라고 하지만 Android에서의 adapter 같은 경우는 어느 레이어계층이 담당해야 할까요? 이미 Android에서의 ListAdpater는 특정 데이터를 UI로 가시화 하는 역할을 잘 담당하고 있는데 말입니다.
제가 MVP패턴을 적용할때 Adapter의 인스턴스를 Presenter 가 가지고 있어야 할지, View 계층이 가지고 있어야 할지 모호 했고 다음과 같은 모호함을 선택 해야 했습니다.

Presenter 에서 인스턴스를 관리 할 경우

  1. Model로부터 Adapter 인스턴스에 Set

  2. Presenter 에서 View에게 알리지 않고 직접 변경

    Activity 에서 인스턴스를 관리 할 경우

Activity,Presenter모두 에서 인스턴스를 관리 할 경우
(생성은 Avtivity , Presenter에게 Adpater 참조값 넘김.)

  • View와 Presenter간 강한커플링이 발생.
  • 응집도 떨어짐. Adapter의 조작이 Presenter, View 모두에서 조작됨.
